## Architecture Overview

ProjetWeb is a microservices application composed of:

| Layer | Component | Technology |
|---|---|---|
| Frontend | `frontend` | Angular 21 / Nginx |
| Gateway | `apigateway` | YARP (.NET 10) |
| Auth API | `authservice` | ASP.NET Core 10 |
| Site Mgmt API | `sitemanagement` | ASP.NET Core 10 |
| Booking API | `bookingservice` | ASP.NET Core 10 |
| Database | `sql` | SQL Server 2025 |
| Migrations | `auth-migrations`, `site-migrations`, `booking-migrations` | EF Core |
| Observability | `aspire-dashboard` | .NET Aspire Dashboard |

Migration services run **once** at startup to apply EF Core database migrations, then exit. All other services remain running.

## Dockerfiles

### Common .NET Build Pattern

All .NET service Dockerfiles follow the same **multi-stage build** pattern:

```
Stage 1 – base      : runtime image (aspnet:10.0)
Stage 2 – build     : SDK image; restore & build
Stage 3 – publish   : SDK image; publish release output
Stage 4 – final     : copy published artefacts into the runtime image
```

Using multi-stage builds keeps the final image small — only the compiled binaries and the ASP.NET runtime are included, not the full .NET SDK.



All .NET images are based on `mcr.microsoft.com/dotnet/aspnet:10.0` (runtime) and `mcr.microsoft.com/dotnet/sdk:10.0` (build).

#### Why shared projects are copied before COPY . .

Each `dotnet restore` step only needs the `.csproj` files it references. Copying them explicitly before the full `COPY . .` allows Docker to cache the restore layer and skip it on subsequent builds when only source files (not project references) change.

---

### ApiGateway

**File:** `ApiGateway/Dockerfile`

```dockerfile
FROM mcr.microsoft.com/dotnet/aspnet:10.0 AS base
USER $APP_UID
WORKDIR /app
EXPOSE 8080
EXPOSE 8081

FROM mcr.microsoft.com/dotnet/sdk:10.0 AS build
ARG BUILD_CONFIGURATION=Release
WORKDIR /src
COPY ["ApiGateway/ApiGateway.csproj", "ApiGateway/"]
COPY ["ProjetWeb.ServiceDefaults/ProjetWeb.ServiceDefaults.csproj", "ProjetWeb.ServiceDefaults/"]
COPY ["ProjetWeb.Shared/ProjetWeb.Shared.csproj", "ProjetWeb.Shared/"]
RUN dotnet restore "./ApiGateway/ApiGateway.csproj"
COPY . .
WORKDIR "/src/ApiGateway"
RUN dotnet build "./ApiGateway.csproj" -c $BUILD_CONFIGURATION -o /app/build

FROM build AS publish
ARG BUILD_CONFIGURATION=Release
RUN dotnet publish "./ApiGateway.csproj" -c $BUILD_CONFIGURATION -o /app/publish /p:UseAppHost=false

FROM base AS final
WORKDIR /app
COPY --from=publish /app/publish .
ENTRYPOINT ["dotnet", "ApiGateway.dll"]
```

**Project dependencies copied:** `ApiGateway`, `ProjetWeb.ServiceDefaults`, `ProjetWeb.Shared`

### Authentication.MigrationService

**File:** `Authentication.MigrationService/Dockerfile`

Migration services follow the same multi-stage pattern with two additional publish flags:

```dockerfile
FROM mcr.microsoft.com/dotnet/aspnet:10.0 AS base
USER $APP_UID
WORKDIR /app

FROM mcr.microsoft.com/dotnet/sdk:10.0 AS build
ARG BUILD_CONFIGURATION=Release
WORKDIR /src
COPY ["Authentication.MigrationService/Authentication.MigrationService.csproj", "Authentication.MigrationService/"]
COPY ["AuthService/Authentication.API.csproj", "AuthService/"]
COPY ["ProjetWeb.ServiceDefaults/ProjetWeb.ServiceDefaults.csproj", "ProjetWeb.ServiceDefaults/"]
COPY ["ProjetWeb.Shared/ProjetWeb.Shared.csproj", "ProjetWeb.Shared/"]
COPY ["ProjetWeb.Shared.Migration/ProjetWeb.Shared.Migration.csproj", "ProjetWeb.Shared.Migration/"]
RUN dotnet restore "./Authentication.MigrationService/Authentication.MigrationService.csproj"
COPY . .
WORKDIR "/src/Authentication.MigrationService"
RUN dotnet build "./Authentication.MigrationService.csproj" -c $BUILD_CONFIGURATION -o /app/build /p:OpenApiGenerateDocuments=false

FROM build AS publish
ARG BUILD_CONFIGURATION=Release
RUN dotnet publish "./Authentication.MigrationService.csproj" -c $BUILD_CONFIGURATION -o /app/publish \
    /p:UseAppHost=false \
    /p:OpenApiGenerateDocuments=false \
    /p:ErrorOnDuplicatePublishOutputFiles=false

FROM base AS final
WORKDIR /app
COPY --from=publish /app/publish .
ENTRYPOINT ["dotnet", "Authentication.MigrationService.dll"]
```

**Notable flags:**
- `/p:OpenApiGenerateDocuments=false` — suppresses OpenAPI doc generation (not needed for a migration-only executable).
- `/p:ErrorOnDuplicatePublishOutputFiles=false` — prevents build errors when multiple project references produce the same assembly filename.

**Project dependencies copied:** `Authentication.MigrationService`, `AuthService`, `ProjetWeb.ServiceDefaults`, `ProjetWeb.Shared`, `ProjetWeb.Shared.Migration`

### ProjetWeb.Frontend

**File:** `ProjetWeb.Frontend/Dockerfile`

```dockerfile
# Build stage
FROM node:22-alpine AS build
WORKDIR /app
COPY package*.json ./
RUN npm ci
COPY . .
RUN npm run build -- --configuration=production

# Production stage
FROM nginx:alpine AS final
COPY nginx.conf /etc/nginx/nginx.conf
COPY --from=build /app/dist/*/browser /usr/share/nginx/html
EXPOSE 8000
CMD ["nginx", "-g", "daemon off;"]
```

- Uses `node:22-alpine` for the build stage and `nginx:alpine` as the final runtime.
- `npm ci` installs exact versions from `package-lock.json` for a reproducible build.
- The Angular production build output is copied into Nginx's HTML root.
- Nginx listens on port **8000** (mapped to host port **4200**).

## docker-compose Files

The docker-compose files live in `ProjetWeb.AppHost/aspirate-output/` and are generated by [Aspirate](https://github.com/prom3theu5/aspirational-manifests) from the .NET Aspire manifest. They are split into two files following Docker Compose convention:

| File | Purpose |
|---|---|
| `docker-compose.yaml` | Service definitions, environment variables, ports |
| `docker-compose.override.yaml` | Build contexts, Dockerfile paths, health checks, `depends_on` conditions |

### docker-compose.yaml

Defines all services with their images, environment variables, and port mappings.

#### aspire-dashboard

```yaml
aspire-dashboard:
  container_name: "aspire-dashboard"
  image: "mcr.microsoft.com/dotnet/aspire-dashboard:latest"
  environment:
    DOTNET_DASHBOARD_UNSECURED_ALLOW_ANONYMOUS: "true"
    ASPIRE_ALLOW_UNSECURED_TRANSPORT: "true"
  ports:
    - target: 18888
      published: 18888   # Dashboard UI
    - target: 18889
      published: 18889   # OTLP gRPC
  restart: unless-stopped
```

The Aspire dashboard collects OTLP telemetry from all services and exposes a web UI at `http://localhost:18888`.

#### sql

```yaml
sql:
  image: "mcr.microsoft.com/mssql/server:2025-latest"
  environment:
    ACCEPT_EULA: "Y"
    MSSQL_SA_PASSWORD: "Znng7j3ORPTvgHAhf00m6e"
    OTEL_EXPORTER_OTLP_ENDPOINT: "http://aspire-dashboard:18889"
    OTEL_SERVICE_NAME: "sql"
  volumes:
    - "data:/var/opt/mssql"
  ports:
    - target: 1433
      published: 14330
  restart: unless-stopped
```

- SQL Server 2025 is used as the single database server hosting three databases (`authdb`, `sitemanagementdb`, `bookingdb`).
- The data volume `data` persists the database files across container restarts.
- Internal port `1433` is published on host port `14330`. All **internal** service-to-service connections use port `1433` via the Docker network.

#### Migration services (auth-migrations, site-migrations, booking-migrations)

Each migration service sets its respective connection string and OTLP endpoint, then exits once migrations are applied:

```yaml
auth-migrations:
  image: "auth-migrations:latest"
  environment:
    ConnectionStrings__authdb: "Server=sql,1433;..."
    OTEL_EXPORTER_OTLP_ENDPOINT: "http://aspire-dashboard:18889"
    OTEL_EXPORTER_OTLP_PROTOCOL: "grpc"
    OTEL_SERVICE_NAME: "auth-migrations"
  restart: unless-stopped   # overridden to "no" in override file
```

#### API services (authservice, sitemanagement, bookingservice, apigateway)

Each API service exposes port `8080` and sends telemetry via gRPC to the Aspire dashboard:

```yaml
authservice:
  image: "authservice:latest"
  environment:
    HTTP_PORTS: "8080"
    ConnectionStrings__authdb: "Server=sql,1433;..."
    OTEL_EXPORTER_OTLP_ENDPOINT: "http://aspire-dashboard:18889"
    OTEL_EXPORTER_OTLP_PROTOCOL: "grpc"
  ports:
    - target: 8080
      published: 10000
```

The API Gateway also receives service discovery environment variables pointing each backend's internal HTTP address:

```yaml
apigateway:
  environment:
    services__authservice__http__0: "http://authservice:8080"
    services__sitemanagement__http__0: "http://sitemanagement:8080"
    services__bookingservice__http__0: "http://bookingservice:8080"
```

#### frontend

```yaml
frontend:
  environment:
    services__apigateway__http__0: "http://apigateway:8080"
    PORT: "4200"
  ports:
    - target: 8000
      published: 4200
```

The Angular SPA is served by Nginx on container port `8000`, mapped to host port `4200`.

### docker-compose.override.yaml

Adds build instructions and dependency ordering on top of the base file.

#### SQL Server health check

```yaml
sql:
  healthcheck:
    test: /opt/mssql-tools18/bin/sqlcmd -S localhost -U sa -P "$$MSSQL_SA_PASSWORD" -C -Q "SELECT 1" || exit 1
    interval: 10s
    timeout: 5s
    retries: 10
    start_period: 30s
```

The health check polls SQL Server every 10 seconds (with a 30-second grace period) before migration services are allowed to start.

#### Build contexts

All service images are built from the repository root (`../../` relative to the compose file location), so the full source tree is available for multi-project builds:

```yaml
auth-migrations:
  build:
    context: ../../
    dockerfile: Authentication.MigrationService/Dockerfile
  restart: "no"

authservice:
  build:
    context: ../../
    dockerfile: AuthService/Dockerfile
```

The frontend is built from its own subdirectory:

```yaml
frontend:
  build:
    context: ../../ProjetWeb.Frontend
    dockerfile: Dockerfile
```

#### Service dependencies

```yaml
auth-migrations:
  depends_on:
    sql:
      condition: service_healthy

authservice:
  depends_on:
    auth-migrations:
      condition: service_completed_successfully

apigateway:
  depends_on:
    authservice:
      condition: service_started
    sitemanagement:
      condition: service_started
    bookingservice:
      condition: service_started

frontend:
  depends_on:
    apigateway:
      condition: service_started
```

> **Note:** `restart: "no"` on migration services means they run once and exit without being restarted. Their `service_completed_successfully` condition ensures the dependent API service only starts after the migration has finished successfully.

## Service Dependency Graph

```
sql (healthy)
├── auth-migrations  (completed_successfully)
│   └── authservice  (started)
│       └── apigateway  (started)
│           └── frontend
├── site-migrations  (completed_successfully)
│   └── sitemanagement  (started)
│       └── apigateway
└── booking-migrations  (completed_successfully)
    └── bookingservice  (started)
        └── apigateway
```

- All migration services wait for SQL Server to be **healthy** before running.
- All API services wait for their corresponding migration service to **complete successfully**.
- The API Gateway waits for all three API services to have **started**.
- The frontend waits for the API Gateway to have **started**.

| Service | Container Port | Host Port | Description |
|---|---|---|---|
| `aspire-dashboard` | 18888 | 18888 | Dashboard Web UI |
| `aspire-dashboard` | 18889 | 18889 | OTLP gRPC ingest |

| `authservice` | 8443 | 10001 | Auth API HTTPS |
| `sitemanagement` | 8080 | 10002 | Site Mgmt API HTTP |
| `sitemanagement` | 8443 | 10003 | Site Mgmt API HTTPS |
| `bookingservice` | 8080 | 10004 | Booking API HTTP |
| `bookingservice` | 8443 | 10005 | Booking API HTTPS |

> **Important:** Internal service-to-service traffic always uses the container port (`8080` for .NET services, `1433` for SQL). The published host ports are only for external access (e.g., from your browser or API client).

## Environment Variables & Configuration

### Connection strings

Connection strings follow the ASP.NET Core naming convention `ConnectionStrings__<name>` and point to the internal SQL Server hostname `sql` on port `1433`:

```
Server=sql,1433;User ID=sa;Password=<password>;TrustServerCertificate=true;Initial Catalog=<database>
```

The same connection is also exposed in several alternative formats for tooling compatibility:
- `*_HOST`, `*_PORT`, `*_USERNAME`, `*_PASSWORD` — individual components
- `*_URI` — `mssql://` URI format
- `*_JDBCCONNECTIONSTRING` — JDBC format
- `*_DATABASENAME` — database name only

### Service discovery

YARP in the API Gateway and Aspire's service discovery use the `services__<name>__http__0` environment variable pattern to locate upstream services:

```
services__authservice__http__0=http://authservice:8080
services__sitemanagement__http__0=http://sitemanagement:8080
services__bookingservice__http__0=http://bookingservice:8080
```

### OpenTelemetry

All services export telemetry to the Aspire Dashboard via gRPC:

```
OTEL_EXPORTER_OTLP_ENDPOINT=http://aspire-dashboard:18889
OTEL_EXPORTER_OTLP_PROTOCOL=grpc
OTEL_SERVICE_NAME=<service-name>
```

Additional .NET-specific OTLP flags are set to enrich spans with exception and event log attributes:

```
OTEL_DOTNET_EXPERIMENTAL_OTLP_EMIT_EXCEPTION_LOG_ATTRIBUTES=true
OTEL_DOTNET_EXPERIMENTAL_OTLP_EMIT_EVENT_LOG_ATTRIBUTES=true
OTEL_DOTNET_EXPERIMENTAL_OTLP_RETRY=in_memory
```

## Observability (OpenTelemetry + Aspire Dashboard)

Every service sends traces, metrics, and logs to the Aspire Dashboard container over OTLP/gRPC on port `18889`. The dashboard aggregates this data and presents it at **http://localhost:18888**.

The dashboard is configured to allow anonymous access and accept non-TLS transport for local development:

```yaml
DOTNET_DASHBOARD_UNSECURED_ALLOW_ANONYMOUS: "true"
ASPIRE_ALLOW_UNSECURED_TRANSPORT: "true"
```

## Data Persistence

A named Docker volume `data` is mounted on the SQL Server container at `/var/opt/mssql`:

```yaml
volumes:
  data: {}
```

This ensures database files survive container restarts and `docker compose down`. To **fully reset** the database, remove the volume:

```bash
docker compose down -v
```

## Running Locally with Docker Compose

> **Prerequisites:** Docker Desktop (or Docker Engine + Compose plugin), at least 4 GB of RAM allocated to Docker.

```bash
# From the repo root
cd ProjetWeb.AppHost/aspirate-output

# Build all images and start all services
docker compose -f docker-compose.yaml -f docker-compose.override.yaml up --build

# Stop and remove containers (keep the data volume)
docker compose -f docker-compose.yaml -f docker-compose.override.yaml down

# Stop and remove containers AND the data volume
docker compose -f docker-compose.yaml -f docker-compose.override.yaml down -v
```

After startup:

| URL | Description |
|---|---|
| http://localhost:4200 | Angular SPA |
| http://localhost:10006 | API Gateway |
| http://localhost:18888 | Aspire Dashboard |
| `localhost,14330` | SQL Server (connect with any SQL client) |

### Startup sequence

1. `sql` starts and passes the health check (~30–60 s on first run).
2. `auth-migrations`, `site-migrations`, `booking-migrations` run in parallel and apply EF Core migrations, then exit.
3. `authservice`, `sitemanagement`, `bookingservice` start up.
4. `apigateway` starts once all three APIs are running.
5. `frontend` starts once the gateway is available.

## Running Locally with .NET Aspire

For the best inner-loop developer experience, run the solution via the `.NET Aspire AppHost` instead of Docker Compose. The AppHost uses the same orchestration logic but runs services as processes (not containers), enables hot reload, and integrates with Visual Studio / VS Code.

```bash
cd ProjetWeb.AppHost
dotnet run
```

The Aspire dashboard opens automatically. All service discovery and connection string injection is handled by the AppHost.

## Aspire AppHost Orchestration

`ProjetWeb.AppHost/AppHost.cs` defines the orchestration graph that is the source of truth for both local Aspire runs and the generated docker-compose files.

Key declarations:

```csharp
// SQL Server with three databases
var sqlServer = builder.AddSqlServer("sql")
    .WithImageTag("2025-latest")
    .WithDataVolume("data")
    .WithEndpoint("tcp", e => { e.Port = 14330; e.TargetPort = 1433; });

var authDb    = sqlServer.AddDatabase("authdb");
var siteDb    = sqlServer.AddDatabase("sitemanagementdb");
var bookingDb = sqlServer.AddDatabase("bookingdb");

// Migration services – run once, then exit
var authMigration    = builder.AddProject<Projects.Authentication_MigrationService>("auth-migrations")
    .WithReference(authDb).WaitFor(authDb);

var siteMigration    = builder.AddProject<Projects.SiteManagement_MigrationService>("site-migrations")
    .WithReference(siteDb).WaitFor(siteDb);

var bookingMigration = builder.AddProject<Projects.Booking_MigrationService>("booking-migrations")
    .WithReference(bookingDb).WaitFor(bookingDb);

// API services wait for migrations to complete
var authApi    = builder.AddProject<Projects.Authentication_API>("authservice")
    .WithReference(authDb).WaitForCompletion(authMigration);

var siteApi    = builder.AddProject<Projects.SiteManagement_API>("sitemanagement")
    .WithReference(siteDb).WaitForCompletion(siteMigration);

var bookingApi = builder.AddProject<Projects.Booking_API>("bookingservice")
    .WithReference(bookingDb).WaitForCompletion(bookingMigration);

// API Gateway waits for all APIs
var apiGateway = builder.AddProject<Projects.ApiGateway>("apigateway")
    .WithReference(authApi).WithReference(siteApi).WithReference(bookingApi)
    .WaitFor(authApi).WaitFor(siteApi).WaitFor(bookingApi);

// Frontend waits for gateway
builder.AddJavaScriptApp("frontend", "../ProjetWeb.Frontend", "start")
    .WithReference(apiGateway).WaitFor(apiGateway)
    .WithHttpEndpoint(port: 4200, targetPort: 4200, env: "PORT", isProxied: false)
    .WithExternalHttpEndpoints();
```

Aspirate (`aspirate generate`) reads the Aspire manifest and produces the `docker-compose.yaml` and `docker-compose.override.yaml` files in `aspirate-output/`. After regenerating those files, review and apply any manual customizations (build contexts, health checks) that are maintained in the override file.
